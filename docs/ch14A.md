## 14주차 A회차: 프로젝트 워크숍 A — 가이드라인 + 개발 시작

> **미션**: 수업이 끝나면 개인 프로젝트 주제를 선정하고 개발 로드맵을 수립하여 14~15주차 동안 배포 가능한 완성도 높은 AI 시스템을 구현할 수 있다

### 학습목표

이 회차를 마치면 다음을 수행할 수 있다:

1. 평가 기준 6가지(기술·성능·완성도·문제정의·발표·코드품질)를 이해하고, 자신의 프로젝트 아이디어가 요구사항을 만족하는지 검증할 수 있다
2. 개인 프로젝트 주제를 선정하고 구체적인 스코프를 정의한 워크시트를 작성할 수 있다
3. 데이터 수집 및 전처리 계획을 수립하고, 대안 데이터 소스를 확인할 수 있다
4. 모델/시스템 아키텍처를 설계하여 8~13주차에서 배운 기술을 선택적으로 조합할 수 있다
5. 개발 로드맵을 14~15주차 2주일 내에 완성할 수 있도록 역산 계획(Backward Planning)하여 수립할 수 있다
6. 제출물 체크리스트(보고서·코드·발표·배포)를 확인하고, 최종 평가 기준과 배점을 이해할 수 있다

### 오늘의 질문 + 배경

**오늘의 질문**: "지난 12주간 배운 모든 기술(Transformer, BERT, LoRA, RAG, Agent, FastAPI)을 조합하여 '실제 사람이 사용할 수 있는' AI 시스템을 2주 안에 만들려면, 어디서부터 시작해야 할까?"

**배경**: 여러분은 이제 LLM의 기초부터 배포까지 전체 생명 주기를 이해했다. 이제 스스로 문제를 정의하고, 기술을 선택하고, 시스템을 구축할 차례다. 단, 시간이 2주뿐이므로 **야심차지만 현실적인** 프로젝트를 설계해야 한다.

### 1부: 평가 기준 및 프로젝트 가이드라인

#### 평가 기준 6가지

개인 프로젝트는 다음 6가지 항목으로 평가된다:

| 항목                                  | 배점 | 설명                                                         | 예시                                                                                                                      |
| ------------------------------------- | ---- | ------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------- |
| **기술적 구현의 깊이**(30점)          | 30%  | 8~13주차 기술을 정당한 이유로 선택하고, 구현이 정확한가      | Attention 메커니즘을 직접 구현했거나, LoRA 파인튜닝 파라미터를 최적화했거나, RAG 검색 성능을 개선했는가                   |
| **모델 성능 및 결과 분석**(20점)      | 20%  | 정량적 평가 지표를 제시하고, 성능이 기준을 만족하는가        | 감정 분류 정확도 85% 이상, 요약문 ROUGE 점수 0.4 이상, 응답 시간 100ms 이하                                               |
| **시스템 완성도**(15점)               | 15%  | 프로토타입이 실제로 작동하는가, 배포되었는가                 | FastAPI로 배포하고 공개 URL 제공, Gradio 웹 인터페이스 구현, Docker 컨테이너화                                            |
| **문제 정의의 명확성과 창의성**(15점) | 15%  | 해결할 실제 문제가 명확한가, 기술적 창의성이 있는가          | "뉴스 기사를 자동으로 요약해주는 시스템"보다 "법률 문서를 한국어/영어로 동시에 분석하고, 쟁점별로 정리하는 시스템"이 높음 |
| **발표 및 의사소통**(10점)            | 10%  | 기술을 명확하고 설득력 있게 전달하는가, 시간 내에 완료하는가 | 5분 안에 문제 정의 → 아키텍처 → 결과 → 한계를 논리적으로 설명                                                             |
| **코드 품질 및 문서화**(10점)         | 10%  | 코드가 재현 가능하고, 문서가 충실한가                        | GitHub README에 설치 방법, 실행 방법, 패키지 요구사항이 명확하고, 주요 모듈에 docstring이 있는가                          |

**배점 해석 팁**:

- 기술(30%) + 성능(20%) = **50점** (모델 중심) — 이 부분이 가장 중요
- 완성도(15%) + 문제정의(15%) = **30점** (시스템·창의성) — "혼자 완성했는가"가 중요
- 발표(10%) + 코드품질(10%) = **20점** (전달·재현 가능성)

**예시 시나리오**:

- 시나리오 A (90점): LoRA로 한국어 감정분류 모델 파인튜닝 → 정확도 88% → FastAPI 배포 → 코드 깔끔 + 명확한 발표
- 시나리오 B (75점): 원론적으로 맞지만 미흡한 완성도 → 아키텍처 좋음 → 배포 안 함 → 테스트 불충분
- 시나리오 C (60점): 기술은 괜찮은데 프로젝트 스코프가 너무 좁음 → "Hugging Face 모델을 API로 호출하기만 했음" → 창의성 부족

#### 프로젝트 아이디어 10가지

다음은 2주 안에 완성 가능한 실제 NLP 프로젝트 예시다. 이 중 하나를 선택하거나, **유사한 범위의 자신의 아이디어**를 제안할 수 있다:

**Group A: 텍스트 분류 + 미니 파인튜닝 (난이도 ★★☆)**

1. **한국어 뉴스 주제 분류기** — KLUE/BERT 파인튜닝으로 뉴스 본문을 5개 카테고리(정치·경제·문화·스포츠·기술)로 자동 분류. 정확도 80% 이상, FastAPI 배포
   - 데이터: Naver 뉴스 API 또는 공개 뉴스 데이터셋
   - 기술: BERT 파인튜닝, LoRA (파라미터 최소화)
   - 결과물: 웹 인터페이스 (텍스트 입력 → 카테고리 + 신뢰도)

2. **리뷰 감정 분석 + 사유 추출** — 식당 리뷰에서 긍정/부정을 분류하고, "맛 좋음", "서비스 나쁨" 같은 이유까지 추출
   - 데이터: 당근마켓, 식신 등의 리뷰 데이터 크롤링 또는 공개 데이터셋
   - 기술: 분류 + Named Entity Recognition (NER), Multi-task Learning
   - 결과물: 리뷰 분석 대시보드 (Gradio)

3. **악의적 댓글 탐지 및 다단계 분류** — 유튜브/Reddit 댓글에서 혐오, 스팸, 광고 등을 자동 분류하고 신뢰도 표시
   - 데이터: 공개 혐오 댓글 데이터셋 (한국어: AbuseFilterCorpus 등)
   - 기술: BERT 멀티클래스 분류, 불균형 데이터 처리 (Focal Loss, 샘플링)
   - 결과물: 배포된 API (신고 시스템에 통합 가능)

**Group B: 텍스트 생성 + RAG (난이도 ★★★)**

4. **법률 문서 자동 요약** — 계약서, 판례문 등 긴 문서를 1-2 단락 요약으로 자동 변환, RAG로 관련 선례 검색
   - 데이터: 공개 법률 데이터베이스 (국가법령정보센터, 판례검색) 또는 합성 데이터
   - 기술: BART/T5 파인튜닝 또는 Claude API + RAG
   - 결과물: 웹 어플리케이션 (PDF 업로드 → 요약 + 관련 선례)

5. **개인화된 학습 Q&A 챗봇** — 학생이 올린 질문에 대해 강의노트/교재에서 자동으로 답변을 찾아 반환 (RAG)
   - 데이터: 대학 강의노트 (구글 드라이브 또는 학교 LMS), 또는 오픈소스 교과서
   - 기술: Embedding (SentenceTransformers) + FAISS 벡터 DB + LLM API (Claude, GPT)
   - 결과물: Discord/Slack 봇 또는 웹 챗 (강의 노트 기반 Q&A)

6. **논문 제목으로부터 초록 자동 생성** — 논문 제목만 입력받아 학술적인 초록(Abstract) 자동 생성
   - 데이터: arXiv, PubMed 등의 논문 + 초록 쌍 크롤링 또는 API
   - 기술: T5/BART 파인튜닝 또는 프롬프트 엔지니어링 (GPT/Claude API)
   - 결과물: 웹 서비스 (제목 입력 → 초록 생성)

**Group C: 고급 기술 (RAG + Agent 또는 멀티모달) (난이도 ★★★★)**

7. **회사 서비스 자동 문의 응답 봇** — 고객 Q&A를 분류 → 자주 묻는 질문(FAQ) DB에서 검색 → 해결 불가능하면 담당자 인수인계. AI Agent 구현
   - 데이터: 회사의 기존 Q&A 로그 또는 합성 데이터
   - 기술: Intention Classification + RAG + LangGraph Agent
   - 결과물: Slack/카톡 연동 챗봇, 성능 메트릭(해결율, 응답시간) 대시보드

8. **논문 자동 분류 및 추천 시스템** — 새로운 논문 제목/초록을 입력받아 분야별(NLP, Vision, RL 등)로 분류하고, 비슷한 기존 논문 5개 추천
   - 데이터: arXiv 메타데이터 (제목, 초록, 카테고리)
   - 기술: Embedding + 벡터 DB (FAISS/Chroma) + Semantic Search
   - 결과물: 웹 서비스 (논문 입력 → 분류 + 관련 논문 추천)

9. **다언어 의료 상담 챗봇** — 증상을 설명하면 가능한 질병 목록을 제안하고, 의료 전문가 대기 기능 포함 (영어, 한국어, 중국어 지원)
   - 데이터: Mayo Clinic, 서울대병원 등의 공개 의료 정보 + 합성 대화 데이터
   - 기술: 다언어 LLM (mT5, 또는 Claude), RAG (의료 지식 베이스), Intent Classification
   - 결과물: 모바일 친화적 웹 인터페이스 (FastAPI + React/HTML)

10. **이미지 캡셔닝 + 질의응답 시스템** — 사진을 업로드하면 사진을 설명하는 한국어 캡션을 자동 생성하고, "이 사진에서 가장 눈에 띄는 것은?"같은 질문에 답변

- 데이터: MSCOCO, Flickr 데이터셋 (또는 자신이 직접 촬영한 사진 500장 이상)
- 기술: BLIP/ViT 모델 (이미지 인코딩) + T5 (캡션 생성) 또는 멀티모달 LLM (GPT-4V, Claude Vision)
- 결과물: 웹 앱 (이미지 업로드 → 캡션 + Q&A)

#### 프로젝트 스코프 제한

**절대 금지 (시간상 불가능)**:

- 대형 모델(7B 이상) Full Fine-tuning → 2주 안에 학습 불가능 (LoRA 또는 프롬프트만 사용)
- 모델 구조 자체 설계 (Transformer 변형 등) → 이론 학습만, 구현은 기존 라이브러리 활용
- 엄격한 학술 평가 (논문 수준의 통계 검증) → 정성적 평가 + 간단한 정량 지표로 충분

**권장 사항**:

- 데이터 크기: 학습 데이터 **500~5000건**, 테스트 데이터 **100건** 수준 (프로토타입 규모)
- 모델: Hugging Face 사전학습 모델 활용 (BERT, GPT-2, T5 등), 또는 Claude/GPT API 호출
- 파인튜닝: LoRA/QLoRA로 파라미터 최소화 (메모리/시간 절감)
- 배포: FastAPI 또는 Gradio (가벼운 웹 인터페이스)

**예상 개발 시간 배분 (14~15주차 2주일 = 20시간 기준)**:

- 데이터 수집/전처리 (4시간)
- 모델 선택 + 파인튜닝 (6시간)
- 아키텍처 설계 + 구현 (5시간)
- 배포 + 테스트 (3시간)
- 보고서 + 발표자료 (2시간)

### 2부: 주제 선정 워크시트 (개별 실습)

이 섹션에서는 프로젝트 주제를 구체적으로 정의한다. 다음 워크시트를 작성하고, 멘토링 또는 피드백을 받아보자.

#### 워크시트 템플릿

```
=== 개인 프로젝트 주제 정의 워크시트 ===

[1단계] 해결할 문제 정의 (1문장)
Q: 이 프로젝트로 무엇을 해결하려는가?
A: 예) "온라인 쇼핑 리뷰를 자동 분석하여 상품 개선점을 추출하는 시스템"

[2단계] 기술 선택 (8~13주차 배운 기술 중 2~3개 선택)
- [ ] BERT (감정분류/NER)
- [ ] GPT (텍스트 생성)
- [ ] LoRA/QLoRA (파인튜닝)
- [ ] RAG (지식 베이스 검색)
- [ ] Agent (다단계 작업)
- [ ] LLM API (Claude/GPT)
- [ ] FastAPI 배포
- [ ] 기타: ___________

[3단계] 데이터 소스 (구체적인 출처 2개 이상)
- 소스 1: ___________ (예: Hugging Face datasets, 공개 GitHub repo, API, 크롤링)
- 소스 2: ___________
- 데이터 크기: 학습 __건, 테스트 __건

[4단계] 평가 지표 (최소 1개, 정량적)
- 지표 1: __________ (예: 정확도, F1 점수, ROUGE, 응답시간)
- 목표값: __________

[5단계] 예상 결과물 (최종 배포 형태)
- [ ] 웹 인터페이스 (FastAPI + HTML)
- [ ] Gradio 데모
- [ ] Discord/Slack 봇
- [ ] 공개 GitHub 저장소
- [ ] 기타: ___________

[6단계] 기술 스택 (언어, 라이브러리, 환경)
Python 3.10+
- 라이브러리: ___________ (예: transformers, torch, langchain, fastapi)
- GPU 필요: [ ] 예 [ ] 아니오
- 배포 환경: ___________ (예: Hugging Face Spaces, 로컬 Docker)

[7단계] 위험 요소 및 대체안
- 위험 요소: ___________ (예: "데이터를 구하기 어려울 수 있음")
- 대체안: ___________ (예: "공개 데이터셋이나 합성 데이터 사용")

[8단계] 개발 로드맵 (14~15주차 일정)
- 14주 B회차: ___________ (예: 데이터 준비, 모델 선택)
- 15주 A회차: ___________ (예: 파인튜닝, 배포)
```

**작성 팁**:

- [1단계]: "뭔가 좋은 시스템을 만들겠다"는 너무 광범위. "누가 / 어떤 문제를 / 어떻게 해결하는가"가 명확해야 함
- [3단계]: "인터넷에서 찾는다"는 불충분. 구체적인 소스, 라이센스, 크기를 미리 확인
- [4단계]: 정성적 평가(예: "작동한다")는 안 됨. 정량 지표 필수 (정확도, 속도, 비용 등)
- [7단계]: "혹시 문제가 있을 수 있음"은 위험 관리 아님. 구체적인 해결책을 미리 준비

### 3부: 아키텍처 설계 가이드

#### 아키텍처 설계의 4단계

프로젝트의 기술 스택을 선택할 때는 다음 4단계를 거친다:

**단계 1: 입력 데이터 형식 결정**

입력이 무엇인지 명확히 해야 모든 것이 결정된다.

| 입력                           | 적합한 기술                                   | 예시                      |
| ------------------------------ | --------------------------------------------- | ------------------------- |
| 텍스트 (문장/문서)             | BERT (분류), T5 (생성), GPT (생성), RAG       | 뉴스 분류, 요약, 질의응답 |
| 구조화된 텍스트 (JSON, 테이블) | Function Calling, Structured Output           | 발주서 → 개인정보 추출    |
| 대화 히스토리                  | Conversational LLM (Claude, GPT), RAG + Agent | 챗봇, Q&A                 |
| 이미지 + 텍스트                | 멀티모달 LLM (GPT-4V, Claude Vision), BLIP    | 이미지 캡셔닝, VQA        |
| 여러 문서 + 질문               | RAG (검색 → 생성)                             | 고객 Q&A, 자동 보고서     |

**단계 2: 처리 방식 선택**

입력 데이터에 따라 처리 방식이 결정된다.

```
[방식 1] 분류 (Classification)
입력 → [모델] → 카테고리 선택
기술: BERT (문장 분류), 이미지 분류 (Vision Transformer)
예: 감정분석, 스팸 탐지, 주제 분류

[방식 2] 생성 (Generation)
입력 → [모델] → 새로운 텍스트 출력
기술: GPT, T5, BART, Claude/GPT API
예: 요약, 번역, 질의응답

[방식 3] 검색 + 생성 (RAG)
입력 → [검색] → 관련 문서 선택 → [생성] → 답변
기술: Embedding + 벡터DB (FAISS, Chroma) + LLM
예: 문서 Q&A, FAQ, 법률 상담

[방식 4] 다단계 작업 (Agent)
입력 → [의도 파악] → [도구 선택] → [실행] → [결과 종합]
기술: LLM + LangGraph + 도구 (검색, 계산, API)
예: 자동 고객 서비스, 리포팅
```

**단계 3: 모델 선택**

각 처리 방식에 따른 구체적 모델:

```
분류:
  - 가벼움: ko-sentence-transformers, DistilBERT
  - 정확함: KLUE/BERT-base, KoBERT
  - 최고 성능: GPT-4, Claude 3 (API)

생성:
  - 한국어 특화: KoGPT2 (구형), Ko-Llama-2
  - 다국어: GPT-2 (기초), mT5, Llama-2
  - 최고 성능: Claude 3, GPT-4 (API 추천)

RAG:
  - Embedding: SentenceTransformers (한국어: KoSimCSE)
  - 벡터DB: FAISS (로컬), Chroma (간편), Weaviate (클라우드)
  - 생성: Claude API 또는 LoRA 파인튜닝 모델

Agent:
  - 프레임워크: LangGraph (추천), LangChain ReAct
  - LLM: Claude 3 (API), GPT-4, Llama-2
```

**단계 4: 배포 선택**

최종 사용자가 접근하는 방식:

```
로컬 프로토타입:
  - Jupyter Notebook / Google Colab
  - 장점: 빠른 개발, 교수와 면담 용이
  - 단점: 타인 접근 불가

웹 인터페이스 (추천):
  - FastAPI + HTML/React
  - Gradio (가장 간편)
  - Streamlit
  - 배포: Hugging Face Spaces (무료), Heroku (유료)

챗봇:
  - Discord Bot / Slack Bot
  - 배포: 개인 서버 또는 클라우드

API:
  - FastAPI 서버
  - 배포: Docker + 클라우드 (AWS, GCP, Azure)
```

[ ] 1. 공개 데이터셋 (즉시 사용 가능)

#### 데이터 전처리 계획 (필수 5단계)

모든 프로젝트는 다음 5단계의 전처리를 거친다:

```
[1단계] 로드 (Load)
└─ 데이터셋을 메모리에 로드
   코드: df = pd.read_csv('data.csv')

[2단계] 정제 (Cleaning)
└─ 결측치, 중복, 특수문자 제거
   예: 텍스트에서 URL, 이모지, 마크다운 제거
       결측치 → 제거 또는 특수값으로 채우기
       중복 행 제거

[3단계] 정규화 (Normalization)
└─ 텍스트를 표준 형식으로 통일
   예: 대소문자 통일 (소문자)
       공백 정규화 (연속 공백 제거)
       특수 토큰 추가 (예: [CLS], [SEP])

[4단계] 분할 (Splitting)
└─ 학습/검증/테스트 세트 분할
   비율: 학습 70~80%, 검증 10%, 테스트 10~20%
   주의: 시간 순 데이터는 시간 순 분할 (섞지 않기)

[5단계] 토크나이저 (Tokenization & Encoding)
└─ 텍스트 → 토큰 ID로 변환
   도구: AutoTokenizer (Hugging Face)
   주의: 모델별 토크나이저 통일 (BERT ≠ GPT)
```

---

#### 데이터 문제 및 해결책

프로젝트 중 자주 만나는 데이터 문제와 해결책:

| 문제                 | 증상                             | 해결책                                                                           |
| -------------------- | -------------------------------- | -------------------------------------------------------------------------------- |
| **불균형 데이터**    | 한 클래스가 다른 것보다 90% 이상 | 언더샘플링 (다수 클래스 제거) 또는 오버샘플링 (소수 클래스 복제) 또는 Focal Loss |
| **결측치**           | NaN 값이 20% 이상                | 열 삭제 또는 평균값/최빈값으로 채우기 (텍스트는 특수토큰 [UNK])                  |
| **길이 편차**        | 텍스트 길이가 10자~10000자       | 최대 길이 설정 (예: 512) 후 절단/패딩                                            |
| **언어 혼재**        | 영어, 한국어, 이모지 섞여있음    | 정규표현식으로 비한국어 제거 또는 다언어 토크나이저 사용                         |
| **노이즈**           | URL, 해시태그, @멘션             | 정규표현식으로 제거: `re.sub(r'https?://\S+', '', text)`                         |
| **클래스 이름 오류** | "긍정", "positive", "pos" 혼재   | 표준화: `label_map = {'긍정': 1, 'positive': 1, 'pos': 1}`                       |

## 정리

14주차 A회차는 **프로젝트 기획의 모든 것**을 다룬다. 이 수업 후:

1. ✅ 프로젝트 주제가 **명확하게 정의**되어 있다

5. ✅ 평가 기준과 제출물이 **명확하다**
