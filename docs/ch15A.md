## 15주차 A회차: 개인 프로젝트 최종 발표

> **미션**: 수업이 끝나면 14~15주차에 걸쳐 개발한 개인 프로젝트를 명확하게 설명하고, 기술적 깊이와 창의성을 보여주며, 동료 피드백을 수용하는 발표를 수행할 수 있다

### 학습목표

이 회차를 마치면 다음을 수행할 수 있다:

1. 자신의 프로젝트 동기와 문제 정의를 과학적이고 설득력 있게 전달할 수 있다
2. 시스템 아키텍처, 데이터 전처리, 모델 설계, 배포 절차를 명확하게 설명할 수 있다
3. 실험 결과를 정량적·정성적으로 분석하고, 인사이트를 도출하며 한계를 객관적으로 평가할 수 있다
4. 기술적 성취와 개선 방향을 균형 있게 제시할 수 있다
5. 동료와 교수의 질문에 명확하고 깊이 있게 답변할 수 있다

## 오늘의 질문 + 발표 의의

**오늘의 질문**: "14주간 배운 NLP 기술의 원리와 구현, 배포를 하나의 완성된 시스템으로 통합할 때, 당신의 접근 방식에서 가장 창의로운 부분은 무엇이었나?"

**발표의 의의**:

개인 프로젝트 최종 발표는 단순한 "과제 결과 보고"가 아니라, **학생 자신이 AI 엔지니어로서 어떤 역량을 갖추었는지를 세상에 보여주는 기회**이다. 이 발표 기록은 포트폴리오가 되고, GitHub 링크와 함께 취업 면접, 대학원 지원, 프리랜서 제안 등에 활용된다. 따라서 다음 세 가지를 의식하며 발표해야 한다:

1. **기술적 깊이**: 당신이 이 프로젝트에서 배운 기술이 실제로 얼마나 깊은지 보여주기
2. **문제 해결 능력**: 개발 중 만난 문제를 어떻게 진단하고 해결했는가
3. **표현 능력**: 복잡한 기술을 다양한 청중(비기술자 포함)이 이해할 수 있게 설명하는 능력

## 발표 구성 및 시간 배분 (5~7분)

개인 프로젝트 발표는 다음 5개 섹션으로 구성된다. 각 섹션의 시간을 정확히 지키면 청중의 이해도가 크게 높아진다.

### 1. 문제 정의와 동기 (1분)

**목표**: 청중이 "이 프로젝트가 왜 존재하는가"를 1분 안에 이해하도록 한다.

**포함 내용**:

- **구체적인 문제 상황**: "어떤 문제가 존재하는가?"
  - 예: "비영어권 언어의 부족한 학습 데이터로 인해 기존 NER 모델의 성능이 30% 낮다"
  - 나쁜 예: "자연어처리는 어렵다" (너무 일반적)
- **기존 해결책의 한계**: "기존 방법이 충분하지 않은 이유는?"
  - 예: "기계 번역으로 데이터를 생성하면 품질이 떨어지고, 도메인 특화 모델은 비용이 크다"
- **당신의 아이디어**: "당신은 어떻게 해결하려고 했나?"
  - 예: "소규모 언어에 강한 다국어 임베딩과 지식 그래프를 결합하여..."

**발표 팁**:

- 처음 30초는 "Hook": 청중의 관심을 사로잡는 구체적 통계나 현실 사례를 제시
- 예: "현재 웹 상의 고품질 NER 데이터의 80%는 영어이고, 한국어 데이터는 2%에 불과합니다"
- 슬라이드는 "문제 상황을 보여주는 1장의 그림"으로 충분

**흔한 실수**:

- "배경 설명에 30초 이상 사용" ❌ → 최대 20초로 제한
- "문제가 모호함" ❌ → 구체적 수치와 사례 제시
- "본인 아이디어가 불명확" ❌ → "우리는 \_\_을 했다"는 명확한 주어와 동사로 시작

### 2. 시스템 아키텍처 및 방법론 (2분)

**목표**: 기술적으로 "어떻게" 구현했는지를 시각적·논리적으로 설명한다.

**포함 내용**:

#### 2.1 전체 파이프라인 아키텍처 (30초)

단계별 데이터 흐름을 블록 다이어그램으로 표현한다:

```
예시 (RAG 시스템):
입력(질문) → 임베딩 → 벡터DB 검색 → 컨텍스트 검색 → LLM 프롬프트 → 생성 → 출력
```

```
예시 (감성 분석 미세조정):
원본 데이터 → 전처리 → 증강 → BERT 로드 → LoRA 설정 → 학습 → 평가 → 배포
```

슬라이드 형식:

- Mermaid 플로우차트 또는 간단한 박스 다이어그램 (1개)
- 각 단계를 1문장으로 설명 (3~4줄)

**발표 팁**:

- "파이프라인의 어디가 당신의 혁신인가"를 강조
- 예: "일반적인 RAG는 단순 벡터 유사도로 검색하지만, 저는 BM25와 의미 검색을 하이브리드로 결합했습니다" (차별성 명시)

#### 2.2 핵심 기술 (1분 15초)

**3~4개의 핵심 기술을 각각 20~30초씩** 설명한다. 각 기술마다:

- **기술 이름** (영문 포함)
- **선택 이유**: "왜 이 모델/기법을 선택했는가"
  - 예: "BERT는 한국어 사전학습이 충분하고, 빠른 추론 속도가 필요했기 때문"
- **주요 파라미터 또는 설정**: 1~2개만
  - 예: "LoRA 계수 r=8, learning rate 2e-4"
- **기대 효과**: "이 선택이 전체 시스템에 어떤 기여를 하는가"
  - 예: "벡터 검색과 BM25의 조합으로 정확도 12% 향상"

슬라이드 형식:

- 기술별 1장씩 (최대 4장)
- 수식 1~2개 (필요시) + 개념도 또는 실행 결과

**발표 팁**:

- 기술을 단순히 "나열"하지 말 것 ❌ → 각각의 "이유와 효과"를 명시
- 깊이 있는 파라미터 1~2개 (예: learning rate schedule, dropout rate)를 언급하면 기술적 신뢰성 증가
- "이 부분에서 가장 어려웠던 점은..." 언급 (문제 해결 과정 보여주기)

#### 2.3 데이터 및 전처리 (15초)

- **사용 데이터**: 데이터셋 이름 및 규모 (샘플 수, 클래스 수 등)
- **전처리 핵심**: 1~2개의 중요한 전처리 단계
  - 예: "문장 정규화, 특수문자 제거, 언어 감지"
  - 나쁜 예: "토큰화를 했습니다" (너무 기본적)
- **데이터 분할**: Train/Val/Test 비율 및 선택 이유

슬라이드 형식:

- 데이터 통계 1개 표 또는 그래프
- 전처리 파이프라인 간단한 흐름도

**발표 팁**:

- "클래스 불균형 문제를 어떻게 해결했는가" 언급 (현실적 문제 해결)
- 데이터 분할이 왜 중요했는지 설명 (예: 시간 기반 분할로 temporal leakage 방지)

**흔한 실수**:

- 아키텍처 설명에 5분 이상 사용 ❌ → 최대 2분
- 너무 상세한 코드 스니펫 ❌ → 핵심 로직만 2~3줄
- "어떻게"만 설명하고 "왜"가 없음 ❌ → 각 선택에 정당성 제시

### 3. 실험 결과 및 데모 시연 (2분)

**목표**: 정량적 성과와 정성적 효과를 동시에 보여준다.

#### 3.1 주요 성과 (1분)

**정량적 평가 (60%)**:

핵심 메트릭 2~3개를 차트 또는 표로 제시한다:

```
예시: 감성 분류 모델
Accuracy: 87.3% (베이스라인 79.2% 대비 +8.1% 향상)
Precision: 0.89, Recall: 0.84, F1-Score: 0.86

[막대 차트: 베이스라인 vs 당신의 모델]
```

```
예시: 문장 요약 모델
ROUGE-1: 0.42 (SOTA: 0.39 대비 +7.7% 향상)
ROUGE-L: 0.38
생성 속도: 2.1초/문장 (GPU)
```

**발표 팁**:

- **차트가 명확한가?** 축 레이블, 범례, 단위를 모두 명시
- **베이스라인과의 비교**: 항상 기존 방법(베이스라인)과 비교하여 상대적 성과 강조
- **통계적 유의성**: "5번 반복 실험의 표준편차 범위: ±0.8%"처럼 신뢰도 명시
- **한계 인정**: "테스트 데이터에서만 유의미한 향상, 실제 배포 환경에서는 미검증"

**정성적 평가 (40%)**:

정량 지표로 잡을 수 없는 질적 개선을 보여준다:

- **에러 분석**: "베이스라인이 실패한 케이스 3가지를 당신의 모델이 맞춘 예시"
  - 예 (NER): "베이스라인은 'SK텔레콤'을 ORG-ORG-PERSON으로 오인식했으나, 당신의 모델은 올바르게 조직명으로 인식"
- **사례 연구**: 3~5개의 실제 예시 (입출력 비교)
- **사용자 피드백**: 만약 있다면, "베타 테스트 사용자의 정성 평가"

슬라이드 형식:

- 메인 메트릭 차트 1~2개 (큼직하게)
- 에러 비교 테이블 또는 사례 3~5개

#### 3.2 데모 시연 (30초) — 선택사항

**라이브 데모가 가능하다면**:

- 입력을 받아 실시간으로 예측 결과를 보여준다
- 예: "웹 UI에 뉴스 기사를 입력하면 주제를 분류하는 데모"
- 예: "챗봇이 사용자 질문에 대답하는 데모"

**데모 팁**:

- **리스크 관리**: 데모는 불확실하므로 "데모 영상 미리 녹화"하여 백업 준비
- **시간 제약**: 데모는 최대 30초 (실패할 경우를 대비)
- **설명과 함께**: "이 시스템은 **를 위해 설계되었고, 실제로 **할 수 있습니다" 설명

**흔한 실수**:

- 메트릭 설명 없이 숫자만 나열 ❌ → "88%라는 것이 무엇을 의미하는가" 설명
- 정량 결과만 제시하고 정성 사례 없음 ❌ → 숫자 + 실제 예시 병행
- "최고 성능을 달성했습니다"라고만 하고 정당성 부족 ❌ → 베이스라인 비교, 실험 설정 명시

### 4. 한계점 및 향후 과제 (1분)

**목표**: 객관적이고 성숙한 기술자로서의 자기평가 능력을 보여준다.

**포함 내용**:

#### 4.1 현재 시스템의 한계 (30초)

**3~4가지 구체적 한계**를 제시한다:

- **데이터 한계**: "학습 데이터가 뉴스 도메인만 포함되어, 소설·학술 텍스트의 성능은 미검증"
- **모델 한계**: "모델이 16개 GPU에서 1시간 학습 필요로, 자원 제약 환경에서는 실용성 낮음"
- **평가 한계**: "테스트 데이터는 모두 2024년 이전 뉴스이므로, 최신 트렌드 용어에 대한 성능은 불명확"
- **영어 이외 언어**: "현재 한국어만 지원하며, 다국어 확장은 미완료"

**발표 팁**:

- "한계를 인정하는 것이 강점"이라는 태도로 임하기
- 한계를 설명할 때 "변명"이 아닌 "사실의 객관적 진술" 투로 말하기
- 한계의 이유를 구체적으로 (예: "예산 부족", "데이터 접근 제약", "시간 부족")

#### 4.2 향후 개선 방향 (30초)

각 한계에 대한 **구체적이고 실행 가능한** 개선 계획을 제시한다:

```
예시:
[현재 한계]                 [개선 방향]                    [기대 효과]
뉴스 도메인만 학습        → 소설·학술 데이터 추가       → 도메인 간 일반화 성능 향상
1시간 학습 필요          → LoRA 경량화 모델로 전환     → 학습 시간 10분 단축
2024년 이전 데이터만 평가 → 2025년 뉴스로 재평가       → 최신 트렌드 적응 확인
한국어만 지원            → 다국어 임베딩 모델 탑재      → 동아시아 3개 언어 지원 가능
```

**발표 팁**:

- 개선 방향은 "할 수 있을 것 같은" 이야기가 아니라 "구체적이고 측정 가능한" 목표로 서술
- "우리는 **을 하면 **%까지 개선할 수 있을 것 같습니다"라고 예측 성능 제시
- 개선이 완료되지 않은 이유를 솔직하게 (시간, 자원 부족)

**흔한 실수**:

- "한계가 없습니다" ❌ → 모든 시스템은 한계를 가짐
- "매우 어렵습니다"만 반복 ❌ → "어려운 이유"와 "해결 방법"을 구체적으로
- "향후 연구 과제입니다"라고만 하고 계획 없음 ❌ → "6개월 내에 \_\_할 예정"처럼 구체화

## 5. 질의응답 (1분)

**목표**: 청중의 질문에 명확하고 깊이 있게 답변하며, 당신의 전문성을 보여준다.

### 질문 예상 영역 및 대비 전략

#### 5.1 기술적 깊이 질문

**자주 나오는 질문**:

- "왜 이 모델을 선택했나? 다른 모델은 안 되나?"
- "하이퍼파라미터는 어떻게 정했나?"
- "이 메트릭을 왜 선택했나? 다른 메트릭은?"
- "수렴 곡선을 보면 과적합이 의심되는데?"

**대비 전략**:

- 발표 전에 "대안 모델 비교"를 미리 분석해두기 (예: BERT vs RoBERTa vs ELECTRA 성능 비교)
- 하이퍼파라미터 선택 이유를 설명할 수 있게 준비 (예: "learning rate를 2e-4로 한 이유는 A, B, C 때문")
- 메트릭 선택의 비즈니스 맥락 설명 (예: "정밀도 중시: 거짓 양성으로 인한 비용이 크므로")

#### 5.2 재현 가능성 질문

**자주 나오는 질문**:

- "코드와 데이터를 공개하나?"
- "학습을 재현할 수 있나?"
- "주어진 하드웨어에서는 어떻게 되나?"

**대비 전략**:

- GitHub 링크 준비 (코드, README 완비)
- `requirements.txt` 작성 (정확한 라이브러리 버전)
- 재현 가이드 마련 (명령어 순서, 예상 소요 시간, 필요 GPU/메모리)

#### 5.3 실무 적용 가능성 질문

**자주 나오는 질문**:

- "실제 프로덕션 환경에 배포할 수 있나?"
- "비용은 얼마나 드나?" (API 호출료, GPU 렌탈료 등)
- "한계를 극복하고 상용화한다면 어떤 순서로?"

**대비 전략**:

- 배포 환경을 미리 설정 (AWS Lambda, Hugging Face Spaces, 또는 자체 서버)
- 비용 추정 계산해두기 (예: "매달 GPU 크레딧 50달러 소요")
- 우선순위 로드맵 수립 (1순위: 정확도 향상, 2순위: 속도 최적화, 3순위: 다국어 지원)

#### 5.4 좋은 답변의 특징

**좋은 답변**:

- "그건 좋은 질문입니다. [구체적 설명] 때문입니다"
- "그 부분은 실제로 저도 고민했던 지점이고, [시도한 방법]을 통해 해결했습니다"
- "그건 현재의 한계인데, 향후 \_\_을 통해 개선할 계획입니다"
- "정확한 수치는 코드에 있으니 GitHub 링크를 참고하시면 됩니다"

**나쁜 답변**:

- "잘 모르겠습니다" ❌ → "현재 검토 중이며, 다음 버전에서 개선할 예정입니다"
- "시간이 부족해서..." ❌ → 객관적 사실 (데이터 부족, 자원 제약)로 설명
- "모델이 똑똑해서 자동으로..." ❌ → 기술적 메커니즘 설명
- "네, 네, 맞습니다"만 반복 ❌ → 근거를 제시하여 답변

**발표 팁**:

- 모든 질문에 답할 필요는 없다. "좋은 질문이지만 지금 설명하기는 너무 상세할 것 같으니, 발표 후 개별로 이야기하겠습니다"가 정중한 대응
- 질문을 받은 후 1~2초 생각하는 시간을 갖기 (성급한 답변은 실수 가능성)

## 발표 슬라이드 템플릿 및 구성 가이드

### 권장 슬라이드 구성 (총 10~15장)

```
1장: 타이틀 슬라이드
     - 프로젝트 제목 (영문 부제 포함)
     - 학번, 이름
     - 발표 날짜

2장: 개요 슬라이드
     - 프로젝트 목표 (1문장)
     - 핵심 기여 (3줄, 불릿)

3장: 문제 정의
     - 문제 상황 (통계 또는 예시)
     - 기존 해결책의 한계
     - (선택) 우리의 접근 방식 미리보기

4장: 시스템 아키텍처
     - 전체 파이프라인 다이어그램
     - 각 단계 1문장 설명

5~8장: 핵심 기술 (각 기술마다 1장)
     - 기술 이름 + 수식 또는 개념도
     - 선택 이유
     - 주요 파라미터

9장: 데이터 및 전처리
     - 데이터셋 통계 표
     - 전처리 파이프라인 또는 예시

10장: 실험 결과 (정량)
      - 메인 메트릭 차트/표
      - 베이스라인 비교

11장: 실험 결과 (정성)
      - 에러 분석 또는 사례 연구 3~5개

12장: 데모 또는 시스템 스크린샷
      - (있다면) 웹 UI 또는 채팅 인터페이스

13장: 한계점 및 향후 과제
      - 현재 한계 3~4개
      - 개선 방향

14장: 결론 슬라이드
      - 핵심 기여 (1문장)
      - 포트폴리오 링크 (GitHub, 배포 URL)
      - 연락처 (선택)

15장: 참고문헌 (선택)
      - 주요 논문/자료 3~5개
```

## 질의응답 운영 가이드

### 질문 유형별 대응 전략

#### 유형 1: 기술 검증 질문

**질문 예시**:

- "BERT의 Transformer 모듈 수를 12에서 6으로 줄여서 실험해 봤는가?"
- "학습 곡선이 완만한데, 언더피팅은 아닌가?"

**대응**:

1. 질문을 명확히 이해하기 (필요시 "다시 말씀드리면..."으로 재확인)
2. 자신이 실험한 것과 하지 않은 것을 구분 명확히
3. "실험하지 않았지만, 향후 시도해 볼 계획입니다"라고 성실하게 응답
4. 관련 논문이나 벤치마크 결과 참고 (예: "선행 연구에서 \_\_% 정도의 성능 저하가 보고되었습니다")

#### 유형 2: 현실 적용성 질문

**질문 예시**:

- "실제 비즈니스에서 사용 가능한 수준인가?"
- "처리 속도는 충분한가?"

**대응**:

1. 현재 상태를 객관적으로 평가 ("프로토타입 수준이며, 프로덕션화까지는 \_\_ 추가 작업 필요")
2. 성능 지표 제시 (응답 시간, 메모리 사용량, 정확도)
3. 개선 경로 제시 ("다음 단계로 모델 경량화를 진행하면 \_%의 속도 향상 예상")

#### 유형 3: 논문/방법론 비교 질문

**질문 예시**:

- "최근 발표된 \_\_과 비교하면 어떤가?"
- "왜 이 논문의 방법을 사용하지 않았는가?"

**대응**:

1. 솔직한 한계 인정 ("해당 논문을 발표 후 알게 되었습니다" 또는 "시간/자원 제약으로 미검토")
2. 현재 방식의 장점 설명 ("우리 방식이 **인 반면, 그 논문은 **이 장점입니다")
3. 향후 비교 계획 언급 ("다음 버전에서 비교 실험을 계획 중입니다")

#### 유형 4: 데이터 관련 질문

**질문 예시**:

- "데이터셋이 편향되지 않았는가?"
- "테스트 데이터로 과적합되지 않았는가?"

**대응**:

1. 데이터 분석 결과 제시 ("클래스 분포 그래프, 도메인별 분포" 등)
2. 편향 완화 방법 설명 ("데이터 증강, 언더샘플링, weighted loss" 등)
3. 검증 방식 설명 ("5-fold cross-validation으로 검증하여 안정성 확보")

#### 유형 5: 도덕/윤리 관련 질문

**질문 예시**:

- "개인정보 보호는 충분한가?"
- "편향된 예측으로 인한 차별은 없는가?"

**대응**:

1. 정중하고 진지한 태도로 답변
2. 자신이 취한 조치 설명 ("데이터는 공개 데이터셋만 사용", "성별/나이별 성능 균형성 검증")
3. 미흡한 부분은 인정하고 개선 계획 제시

### 질의응답 에티켓

**발표자 측**:

- 질문을 끝까지 듣고 중간에 끼어들지 않기
- 질문자를 존중하는 태도 ("좋은 질문입니다", "흥미로운 지적입니다")
- 답변 후 "설명이 충분했는지" 확인 ("제 답변이 도움이 되었는지요?")

**청중 측**:

- 질문은 건설적이고 명확하게
- "당신의 방법이 최고입니까?" 같은 과도한 질문은 피하기
- 답변 후 감사 표현

## 좋은 발표와 나쁜 발표의 구체적 사례

### 사례 1: 감성 분석 모델 파인튜닝

**나쁜 발표 예시**:

```
(1분차) "저는 BERT를 써서 감성 분석을 했습니다.
(2분차) BERT는 구글에서 만든 모델이고...
        (배경 설명에 너무 많은 시간 사용)
(3분차) 저는 데이터를 전처리했습니다.
        (구체적 내용 없음)
(4분차) 정확도가 85%입니다.
        (베이스라인과의 비교 없음)
(5분차) 음... 질문이 있습니까?
        (한계점, 향후 과제 없음)
```

**좋은 발표 예시**:

```
(30초) "네이버 영화 리뷰 31만 건을 분석했을 때,
        기존 감성 사전은 신조어(예: '꿀잼')을 인식 못해
        성능이 73%에 그칩니다.
        저는 문맥을 반영한 BERT 파인튜닝으로 이를 해결하려 합니다."

(45초) "전체 파이프라인은:
        데이터 전처리 → 신조어 확장 → BERT 로드 → LoRA 어댑터 추가 →
        Trainer로 학습 → 평가 → FastAPI로 배포입니다."

(50초) "핵심은 LoRA 어댑터입니다.
        전체 모델은 110M 파라미터지만,
        LoRA 계수 r=8로 설정하면 1.6M 파라미터만 학습하면서
        정확도 손실은 단 0.5%입니다."

(40초) "실험 결과, 정확도가 73% → 85.6%으로 향상되었고,
        [차트] 특히 신조어를 포함한 테스트셋에서
        기존 모델(78%) vs 저희 모델(86.2%)의 차이가 명확합니다."

(30초) "정성적으로는, 기존 모델이
        '진짜 꿀잼이다' → 긍정 으로만 분류했다면,
        제 모델은 '진짜'(강도 증폭)를 반영해
        신뢰도 0.95로 긍정 분류합니다."

(30초) "현재의 한계는:
        1) 학습 데이터가 네이버 리뷰만 포함 → 카카오, 쿠팡 데이터 추가 필요
        2) 7시간 학습 필요 → 경량 모델(DistilBERT)로 재학습해 3시간 단축 예정
        3) 반의어 구분 미흡 (예: '별로임'을 긍정으로 오인식) →
           hard negative mining으로 개선 계획"

(1분) 질의응답
```

**차이점 분석**:

| 항목         | 나쁜 발표          | 좋은 발표                            |
| ------------ | ------------------ | ------------------------------------ |
| 문제 정의    | 모호 ("감성 분석") | 구체적 (신조어 미인식 73% 정확도)    |
| 솔루션 설명  | "BERT를 썼습니다"  | "LoRA r=8로 1.6M 파라미터만 학습"    |
| 결과 제시    | "85% 정확도"       | "73% → 85.6%, 베이스라인 78% 대비"   |
| 정성 분석    | 없음               | 예시 3개 (신조어, 강도 표현, 반의어) |
| 한계 및 향후 | 없음               | 3가지 한계 + 구체적 개선 계획        |

## 발표 후 피드백 반영 및 포트폴리오 구성

### 발표 후 재정리 (발표 1주일 내)

발표를 마친 후 다음을 진행한다:
